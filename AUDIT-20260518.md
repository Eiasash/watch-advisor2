### FIXED — MEDIUM (flaky/broken test, Windows-only): `tests/authSubscriptionUniqueness.test.js`
- The guard test (added in the v1.13.47 head commit) asserts that only `authStore.js` / `supabaseAuth.js` / `authedFetch.js` may subscribe to Supabase auth events.
- Bug was **in the test, not the source**: `path.relative()` returns backslash-separated paths on Windows (`src\stores\authStore.js`), but the `ALLOWED` set holds forward-slash entries (`src/stores/authStore.js`). `ALLOWED.has(rel)` was always `false` locally, so the three explicitly-allowed files were reported as offenders. The reported offenders were *exactly and only* the three ALLOWED files — zero real source violations.
- Passes on Linux CI (forward slashes), failed only on Windows. Genuine non-portable test.
- Fix: `relative(...).split(sep).join("/")` before the `ALLOWED.has()` check; added `sep` import. Minimal, no source change. Test now passes; full suite green.
- Commit: `13ae5b4` `fix(test): normalize path separators in authSubscriptionUniqueness guard`

Per hard rule "Engine invariants — do NOT change them", I left the code untouched. **Action for Eias:** update the canonical invariant list / audit-prompt template so future audits don't flag this. No code change needed or wanted.

### 2. `node node_modules/.bin/vitest run` does not work on Windows
The prompt's exact test command (`node node_modules/.bin/vitest`) fails — `.bin/vitest` is the bash-shim, not JS; native node parses the shell script and throws `SyntaxError`. Working invocation on this Windows box: `node node_modules/vitest/vitest.mjs run` (the real JS entrypoint the shim ultimately calls). All test runs in this audit used the `.mjs` form. **Action for Eias:** future overnight-audit prompts for Windows should specify `node node_modules/vitest/vitest.mjs run` (or just `npm test`).

### 3. 10 no-op `eslint-disable` comments (deferred — cosmetic only)
